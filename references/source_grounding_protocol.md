# Source Grounding Protocol

本协议用于保证生成的课程总览和模块知识手册准确、可追溯、详略有序。

## 1. 来源优先级

当材料之间存在冲突时，优先级建议如下：

1. 教师发布的期末复习提纲、考试大纲、样题。
2. 课程主课件 PDF/PPT。
3. 实验指导书、作业讲义、补充材料。
4. 模型自己的背景知识。

模型背景知识只能用于辅助解释，不能冒充课件内容。

## 2. 内容来源标注

模块文档应尽量包含来源定位表：

| 内容 | 来源文件 | 页码/幻灯片 | 图片路径 | 复核状态 |
|---|---|---|---|---|
| 两阶段提交协议流程图 | chapter_04_transactions.pdf | p.12-p.18 | images/module02_slide14_2pc_flow.png | 已回读并插图 |

如果无法获得页码或幻灯片编号，至少标注文件名和章节标题。

## 3. 图片来源保留要求

课件中的重要结构图、流程图、架构图、状态图、复杂表格、公式推导图、示例运行截图等，应作为原始视觉材料保留下来。不能只用文字转述代替。

处理要求：

- 能导出原图时，优先导出原图。
- 无法导出原图但能查看页面/幻灯片时，使用截图或裁剪保留关键区域。
- 图片保存到 `images/`，并使用稳定文件名，例如 `module02_slide14_2pc_flow.png`。
- 模块 Markdown 中必须使用 `![图示名称](images/<文件名>.png)` 插入图片。
- 图片下方继续说明来源、对应知识点、读图方法和考试提示。
- 如果确实无法提取或截图，应写明“图示待补”、原始位置和原因。

## 4. 模块回读要求

在写每个模块前必须回读相关材料。

回读时检查：

- 定义是否准确。
- 术语是否与课件一致。
- 公式是否完整。
- 算法步骤是否缺失。
- 图示是否被错误归类。
- 重要图示是否已经保存为图片并插入 Markdown。
- 图片路径是否与 `images/` 中的文件一致。
- 例子是否能对应到正确知识点。
- 是否有老师标注的重点、难点、注意事项。

## 5. 详略控制

按如下原则控制详略：

| 内容类型 | 处理方式 |
|---|---|
| 课程主线 | 在总览中讲清楚 |
| 核心概念 | 在模块手册中清楚解释 |
| 关键机制/算法/公式 | 详细拆解 |
| 例子/图示 | 单独列出，保留图片，并解释 |
| 补充阅读内容 | 简略说明或标记为扩展 |
| 与考试关系弱的细节 | 可合并或略写 |

## 6. 不确定性处理

如果无法确认某个知识点是否属于课程重点，应写：

```markdown
> 说明：该考点的重要性是根据课件中出现频率和模块位置推断的，未发现明确考试大纲标注。
```

如果材料中没有支持某个知识点，应避免写成课程要求。可以写：

```markdown
> 补充理解：以下内容不是课件中的明确表述，只用于帮助理解。
```

## 7. 禁止行为

禁止：

- 只根据文件标题生成完整内容。
- 只根据第一遍总览生成模块手册。
- 编造页码、图号或来源。
- 把通用教材知识误写成课件重点。
- 忽略原课件中的图、表、例子、公式。
- 在可以提取或截图时，只用文字描述替代重要课件图片。
